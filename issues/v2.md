# Strata v2 — Backlog

Items discovered during QA review that are out of scope for the current beta but should be tracked for v2.

---

## Features (new user-facing capabilities)

### Transaction UI
- `Transaction` type is declared in the frontend (`lib/types/index.ts`) and the backend schema has a full `transactions` table, but there is zero UI for it: no list, no create/edit/delete form, no entry in the sidebar
- Blocking: backend endpoints for transactions don't exist yet
- Scope: full CRUD feature (acquire, dispose, adjust) with impact on asset history

### Category + Tag Rename/Edit
- Once a category or tag is created, it can only be deleted (losing all asset associations)
- Need: `PUT /api/v1/categories/:id` and `PUT /api/v1/tags/:id` endpoints
- Frontend: edit form in CategoriesPage and TagsPage

### Asset Snapshot Delete
- Users cannot delete an incorrectly-entered snapshot (wrong value, typo)
- Need: `DELETE /api/v1/assets/:id/snapshots/:snapshotId` endpoint
- Frontend: delete button per row in AssetSnapshotsList

### Data Factory Reset (Settings)
- No "Delete all data" or "Reset to empty" in the Settings page
- Useful for starting fresh after testing with demo data
- Consider: either a backend endpoint or a docker:reset-equivalent for production data

## API Improvements

### GET /portfolio-snapshots/:id
- `PortfolioSnapshotService.findById()` exists and works internally (used by delete)
- Just needs a controller route exposing it
- Low effort: single `@Get(':id')` endpoint

### ISO 4217 Currency Format Validation
- `CreatePortfolioSnapshotDto.currency` accepts any string (`"MONOPOLY_MONEY"` is valid)
- Fix: add `@Length(3, 3)` or `@IsIn(['EUR', 'USD', 'GBP', ...])` to the DTO
- Low effort once the currency list is decided

### computeCurrentValue Query Optimization
- Current: `findAll()` loads ALL assets with ALL their snapshots, transactions, tags, categories just to pick the latest snapshot per asset
- Fix: use `findLatestByAsset()` per non-disposed asset, or a single aggregation query
- Impact: irrelevant with <100 assets, but technically wasteful

## Quality / Infrastructure

### Pagination on List Endpoints
- All `findAll()` endpoints return unbounded results with full eager-loaded relations
- Acceptable for a personal tracker (<100 assets) but fragile at scale
- `GET /assets` is the most expensive: loads all snapshots, transactions, tags, categories for every asset
- Solution: cursor-based pagination with `limit/cursor` params; or at minimum, lazy-load relations

### Database Indexes
- SQLite does not auto-index foreign key columns
- Missing indexes on high-frequency columns: `asset_snapshots(asset_id)`, `asset_snapshots(observed_at)`, `transactions(asset_id)`, `portfolio_snapshots(observed_at)`, `asset_categories(category_id)`, `asset_tags(tag_id)`
- Fix: new Prisma migration with `@@index` declarations
- Impact: negligible for <100 assets, noticeable at scale

### Backup Merge Mode E2E Test
- Both backup e2e tests use `mode: 'replace'`
- Merge semantics (no wipe, upsert behavior, FK conflict handling) are tested with mocked Prisma only
- A real SQLite-backed e2e test would catch conflicts that the mock can't simulate
